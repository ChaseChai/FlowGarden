## 三、技术架构

```
业务脚本
  │
  │  get_system_info() / download_audio_file() / wait_sensor_data() ...
  ▼
高层功能函数
  │
  ▼
RingSoundClient
  │  请求与响应匹配、命令队列、主动事件分发
  ▼
PacketStream + BinaryReader/BinaryWriter
  │  BLE 分片重组、协议头、CRC、网络字节序字段
  ▼
NusClient
  │  bleak 扫描、连接、写特征值、订阅通知
  ▼
戒指 Nordic UART Service (NUS)
```

### 3.1 BLE 传输层（Nordic UART Service）

| 用途 | UUID | 数据方向 |
|------|------|----------|
| Service | `6E400001-B5A3-F393-E0A9-E50E24DCCA9E` | 服务标识 |
| TX Characteristic | `6E400003-B5A3-F393-E0A9-E50E24DCCA9E` | 戒指通知 Python |
| RX Characteristic | `6E400002-B5A3-F393-E0A9-E50E24DCCA9E` | Python 写入戒指 |

SDK 按 MAC 地址扫描和连接设备，不依赖广播设备名。

### 3.2 协议包重组

- BLE 一次通知不等于一个完整业务协议包
- `PacketStream.feed()` 持续缓存 BLE 字节，按 magic 和 `body_length` 提取完整协议包
- 协议头固定 11 字节，包体超过 5120 字节、CRC 不正确或包头不合法时抛出 `ProtocolError`

### 3.3 v4 协议包格式

| 偏移 | 字段 | 长度 | 字节序 | 说明 |
|------|------|------|--------|------|
| 0 | `magic_number` | 1 | 无 | 固定 `0x3F` |
| 1 | `version` | 2 | 大端 | SDK 发送 4，接受 ≤4 |
| 3 | `command` | 2 | 大端 | 例如 `0x0101`、`0x0505` |
| 5 | `body_length` | 4 | 大端 | 包体字节数（不含 11 字节包头） |
| 9 | `body_crc` | 2 | 大端 | 包体 CRC16；空包体为 0 |
| 11 | `body` | 可变 | 按命令定义 | 命令包体 |

> 注：录音 `.bin` 内部的 Speex 帧长度使用 2 字节小端序（例外）

## 四、核心功能

### 4.1 连接与会话管理

| 函数 | 说明 |
|------|------|
| `scan_rings()` | 扫描附近 BLE 设备，可按 MAC 地址过滤 |
| `connect_ring()` | 创建并连接 `RingSoundClient` |
| `RingSoundClient` | 高层连接对象，负责连接、收包重组、命令请求、事件等待 |

### 4.2 系统信息与日志

| 函数 | 说明 |
|------|------|
| `get_system_info()` | 读取固件版本、时间、存储、电量、SN、CPUID、型号 |
| `get_log_storage()` | 查询日志存储信息 |
| `read_log_chunk()` | 按日志索引、偏移、长度读取日志 |

### 4.3 校时服务

| 函数 | 说明 |
|------|------|
| `enable_time_sync()` | 注册 `0x0401` 自动应答处理器，设备请求时自动回复 |
| `send_time_response()` | 手动发送校时应答 |

### 4.4 录音下载与解码

| 函数 | 说明 |
|------|------|
| `get_audio_file_count()` | 查询设备中保存的录音数量 |
| `download_audio_file()` | 完整下载指定录音（默认 quick 链路） |
| `receive_auto_audio_file()` | 接收录音保存后设备主动上报的连续 `0x0505` |
| `clear_audio_files()` | 删除设备内所有录音（破坏性操作） |
| `save_audio_bundle()` | 保存原始录音并生成可播放 WAV |
| `decode_audio_to_wav()` | 把录音字节解码成 WAV 字节 |

**录音传输链路**：
1. **quick 链路**（推荐）：`0x0509 → 0x0504 → 0x0505...`，缺帧时 `0x0506` 补帧
2. **普通链路**：`0x0503 → 0x0504`，循环 `0x0506 → 0x0505`，最后 `0x0507 → 0x0508`
3. **保存后自动上报**：录音保存成功后连续 `0x0505`，无预先 `0x0504`

### 4.5 IMU 与动作事件

| 函数 | 说明 |
|------|------|
| `start_sensor_report()` | 发送 `0x0601`，开启 `0x0605` BLE 上报（设备需已处于手势模式） |
| `wait_sensor_data()` | 等待批量 IMU 数据 `0x0605` |
| `stop_sensor_report()` | 发送 `0x0603`，关闭实时上报 |
| `wait_sensor_double_tap_event()` | 等待 `0x0701` 普通双击事件 |
| `wait_sensor_gesture_event()` | 等待 `0x0702` HMM 手势事件 |
| `wait_sensor_key_double_press_event()` | 等待 `0x0703` 按键双击事件 |
| `wait_sensor_key_single_press_event()` | 等待 `0x0704` 按键单击事件 |

### 4.6 异常类型

| 异常 | 说明 |
|------|------|
| `RingSoundError` | SDK 异常基类 |
| `TransportError` | BLE 传输异常（未安装 bleak、设备未找到、连接失败等） |
| `ProtocolError` | 协议异常（包头、CRC、包体长度、字段解析错误） |
| `TimeoutError` | 等待设备响应或主动上报超时 |
| `DeviceError` | 设备返回非 0 错误码（busy、参数错误、文件不存在等） |
| `AudioDecodeError` | 录音数据无法解码为可播放 WAV |
| `SpeexDecoderUnavailable` | 缺少 ffmpeg 或自定义 Speex 解码器不可用 |

## 五、音频处理流程

### 5.1 设备端录音编码

```
PDM/双声道 PCM (16000Hz, 16bit, 左右交错, 100ms/块)
  → 提取左声道，增益处理（left * 2）
  → 单通道 PCM (16000Hz, 16bit, 1600采样点/块, 3200字节)
  → Speex Wideband 编码（FRAME_SIZE=320, 质量=3, 复杂度=3）
  → 100ms PCM 块编码为 5 个 Speex 帧
```

### 5.2 设备保存的 Speex 数据格式

```
[2字节小端长度][Speex payload][2字节小端长度][Speex payload]...
```

例如 payload 长度为 20 字节时，前两个字节为 `14 00`（小端序，0x0014 = 20）。

### 5.3 SDK 解码流程

```
下载的 .bin
  → 解析 2 字节小端长度前缀
  → 得到独立 Speex payload
  → SDK 构建 Ogg Speex 容器
  → ffmpeg 解码为 s16le PCM
  → SDK 添加 WAV 文件头
  → 保存 .wav
```

### 5.4 音频格式对比

| 名称 | 是否压缩 | 是否自带播放参数 | 当前用途 |
|------|----------|------------------|----------|
| 设备 `.bin` | 是 | 否 | 原始长度前缀 Speex 流 |
| 裸 Speex payload | 是 | 否 | 单个编码帧 |
| Ogg Speex `.spx`/`.ogg` | 是 | 是 | ffmpeg 中间容器 |
| PCM `s16le` | 否 | 否 | ffmpeg 解码后的原始采样 |
| WAV `.wav` | 否 | 是 | PCM 加 RIFF/WAV 头，适合播放 |

## 六、设备交互模式

### 6.1 设备工作模式

设备维护**录音模式**和**手势模式**两个工作模式：

| 模式 | 默认状态 | IMU 采集 | 主要功能 |
|------|----------|----------|----------|
| 录音模式 | 启动后默认 | 关闭并被阻止 | 录音保存后主动上报 `0x0505` |
| 手势模式 | 单击切换进入 | 开启 | 实时 IMU 上报 `0x0605`、HMM 手势识别 `0x0702` |

**关键限制**：SDK 不能主动查询或切换录音/手势模式。

### 6.2 按键状态机

| 用户操作 | 设备处理 | 协议事件 | 模式影响 |
|----------|----------|----------|----------|
| 单击（<200ms） | 50ms 消抖，500ms 双击窗口 | `0x0704` | 窗口结束后尝试切换模式 |
| 双击 | 第二次有效短按在 500ms 内 | `0x0703` | 不切换模式 |
| 长按（>300ms） | 根据当前模式开始操作 | 无独立事件 | 录音模式开始录音；手势模式开始 HMM 手势会话 |
| 长按松开 | 结束当前录音或手势会话 | 录音数据或 `0x0702` | 模式本身不变化 |

### 6.3 LED 状态指示

| 触发条件 | 灯效 | 开发者理解 |
|----------|------|------------|
| 录音进行中 | 绿灯常亮 | 正在录音 |
| 手势会话进行中 | 红灯常亮 | 正在采集手势 |
| 模式切换/启动失败 | 红灯闪烁 3 次（100ms 亮/灭） | 本次操作失败 |
| 正在充电 | 红灯 500ms 亮/灭 | 充电中 |
| 充电完成 | 绿灯常亮 | 已充满 |

## 七、公开 API 速查

### 7.1 连接与会话

```python
import ring_sound as sdk

# 扫描设备
devices = await sdk.scan_rings(address="F1:C1:8A:35:40:FB")

# 连接设备（推荐使用 async with）
async with sdk.RingSoundClient(address=ADDRESS) as ring:
    info = await sdk.get_system_info(ring)
```

### 7.2 系统信息

```python
info = await sdk.get_system_info(ring)
print(info.firmware_version, info.battery_percent, info.audio_storage_available)
```

### 7.3 自动校时

```python
sdk.enable_time_sync(ring)
# 设备请求时 SDK 自动回复
```

### 7.4 下载录音

```python
count = await sdk.get_audio_file_count(ring)
if count:
    info, raw_audio = await sdk.download_audio_file(ring, file_index=0)
    bundle = sdk.save_audio_bundle(
        file_index=info.file_index,
        data=raw_audio,
        metadata={"record_time": info.record_time},
        output_dir="audio",
    )
```

### 7.5 接收刚结束的录音

```python
file_index, raw_audio = await sdk.receive_auto_audio_file(ring, timeout_s=60.0)
bundle = sdk.save_audio_bundle(file_index=file_index, data=raw_audio, output_dir="audio")
```

### 7.6 批量 IMU 数据

```python
# 前提：设备已处于手势模式
await sdk.start_sensor_report(ring)
try:
    batch = await sdk.wait_sensor_data(ring, timeout_s=5.0)
    for sample in batch.samples:
        print(sample.timestamp_ms, sample.accel_x, sample.gyro_x)
finally:
    await sdk.stop_sensor_report(ring)
```

### 7.7 手势事件

```python
event = await sdk.wait_sensor_gesture_event(ring, timeout_s=30.0)
print(event.gesture_id, sdk.sensor_gesture_name(event.gesture_id))
# 手势 ID: 0=idle, 1=rotate_back, 2=rotate_front, 3=wave
```

## 八、设备模式与关键注意事项

### 8.1 模式切换限制

- SDK **无法主动查询或切换**录音/手势模式
- `0x0704` 只确认设备识别到按键单击，**不携带切换后的模式**
- 设备忙碌或长按收尾阶段时，即使收到 `0x0704`，**也不能断言模式切换成功**
- 长按松开后 250ms 内，设备会忽略短按触发的切换请求

### 8.2 录音与 IMU 互斥

- 录音模式下本地 IMU 被关闭并阻止，`start_sensor_report()` 会返回错误码 2
- 必须通过有效按键单击切换到手势模式后，才能成功开启 IMU 上报

### 8.3 自动上报与主动下载互斥

- `receive_auto_audio_file()` 不能与 `download_audio_file()`、`read_audio_frame()` 并发运行
- 它们消费同一个 `0x0505` 队列，可能互相取走数据

## 九、常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `.bin` 下载成功但无法播放 | `.bin` 是长度前缀 Speex 流，不是播放器通用容器 | 使用 `save_audio_bundle()` 生成 WAV |
| ffmpeg 报输入格式错误 | 数据混入协议字段或 BLE 分片未重组完成 | 确认数据从第一个 Speex 长度字段开始 |
| 解码时长与录音时长不一致 | 丢帧、重叠帧、错误 `frame_offset` | 优先解析 2 字节小端长度前缀 |
| 收到一组没有 `0x0504` 的 `0x0505` | 当前固件录音保存后自动发送 | 使用 `receive_auto_audio_file()` 接收 |
| `start_sensor_report()` 返回设备忙碌 | 设备处于录音模式，IMU 被关闭 | 先让用户单击切换到手势模式 |

1. NUS UUID、协议版本、11 字节包头和 CRC 是否变化

8. 按键消抖、短按、长按、双击窗口和长按收尾抑制时间
9. 录音/手势模式的进入条件、IMU 状态和 `0x0605` 上报开关

## 十一、总结

本工作空间包含一套完整的**语音戒指 BLE 通信与数据处理 SDK**，涵盖：

1. **通信层**：基于 Nordic UART Service 的 BLE 通信，支持协议包重组、CRC 校验、命令队列管理
2. **业务层**：系统信息、日志、校时、录音下载、IMU 数据采集、动作事件识别
3. **音频处理层**：Speex 解码、Ogg 容器构建、ffmpeg 集成、WAV 输出
4. **机械设计层**：4 组戒指打印模型（STEP 格式），支持外壳和按键的 3D 打印

SDK 采用单文件设计（`ring_sound.py`），便于集成和维护。文档体系完整，包括技术说明、调用手册和协议文档，适合不同角色的开发者查阅。
